## 📂 公開プロジェクト一覧

現在、GitHub でいくつかの完成したプロジェクトを公開しています。以下はその一部です：

### ツールと統合系

- [**AutoTraderX**](./autotraderx/index.md)：

  これは私が台湾の証券取引業者のシステムと連携する練習をしたときの記録です。現在は「元富証券」の API を探索していますが、今後は「富邦証券」を探索する予定ですが、まだ時間がありません。

  :::tip
  開発の感想を聞かれたら、おそらく「心の余裕がない」でしょう。😓
  :::

  ***

- [**Capybara**](./capybara/index.md)：

  ここでは、コンピュータビジョンの分野でよく使用される構造、例えば `Boxes`、`Polygons`、`Keypoints` などが定義されています。

  さらに、画像処理（opencv）、モデルアーキテクチャ（pytorch）、推論ツール（onnxruntime）、および環境設定に関する内容も含まれており、これらは私たちが仕事でよく使用するツールです。

  ***

- [**DocsaidKit（近日廃止予定）**](./docsaidkit/index.md)：

  一定期間の使用後、このツールキットを分割することに決めました。PyTorch に関連するトレーニングツールを削除し、モデル推論と画像処理に関連する機能のみを残しました。

  最終的にこのプロジェクトは三つに分割されました：

  - [**Capybara**](https://github.com/DocsaidLab/Capybara)：モデル推論と画像処理に関連する機能。
  - [**Chameleon**](https://github.com/DocsaidLab/Chameleon)：純粋な PyTorch トレーニングツール。
  - [**Otter**](https://github.com/DocsaidLab/Otter)：PyTorch-Lightning ベースのトレーニングツール。

  :::tip
  これらのパッケージ名について質問があるかもしれませんが、何かおかしな名前だと思っていませんか？🤔🤔🤔

  そんなことはありません！今では大手機関が発表する論文もみんなこんな風に奇妙な名前をつけています。私たちはその偉大な方々に敬意を表しています！
  :::

  ***

- [**GmailSummary**](./gmailsummary/index.md)：

  これは私が Gmail と OpenAI を接続する練習をしたときの記録です。内容は、Google と OpenAI の API の更新後に動作しなくなる可能性があります。

  このプロジェクトは以前数ヶ月動作していましたが、OpenAI への支払いが終了したため、現在は停止しています。

  ***

- [**WordCanvas**](./wordcanvas/index.md)：

  以前、合成訓練データを開発するために基本的な機能を抽象化し、ツールにまとめたプロジェクトです。このプロジェクトの主な機能は、フォントファイルを画像にレンダリングすることです。

### 深層学習プロジェクト

- [**DocAligner**](./docaligner/index.md)：

  これはドキュメントアライメントのプロジェクトで、ドキュメントの 4 つの角を検出する機能を持っています。

  この機能は単純ですが、多くのアプリケーションシーンで役立ちます。現在は 4 つの角を検出する機能だけですが、時間があれば追加の機能を加える予定です。

  ***

- [**DocClassifier**](./docclassifier/index.md)：

  これはドキュメント分類のプロジェクトで、ドキュメントを異なるカテゴリーに分類する機能を持っています。

  このプロジェクトではトレーニングモジュールも公開しており、私の各モデルプロジェクトは同じ構築ロジックを使っています。もし他のモデルに興味があれば、このプロジェクトを参考にして、自分専用のトレーニング環境を構築することができます。

  ***

- [**MRZScanner**](./mrzscanner/index.md)：

  これはドキュメント上の MRZ エリアを識別する機能です。

  最初は End-to-End のモデルを作ろうと考えましたが、最終的には期待通りの結果にはならなかったものの、小さな成果があったので、これを整理してオープンソースプロジェクトとして公開しました。必要な方に役立てばと思います。

しかし、私の能力には限りがあり、翻訳作業を自分だけで行うことができません。そのため、市場にあるさまざまな `GPTs` に手伝ってもらっています。

標準的な作業フローは、各記事の段落を取り、それを直接 `GPTs` に翻訳させるというものです。翻訳結果を受け取った後は、人工で校正を行い、目視で確認できる誤りを除去します。

## 🔄 モデルの調整

これはおそらく最も関心のあるテーマでしょう。

私が定義したテーマに基づいて提供したモデルを使用すれば、ほとんどのアプリケーションシーンは解決できると考えています。

ですが、いくつかのシーンでは、より良いモデル結果が必要になる場合があるため、データセットを収集してモデルの微調整を行う必要があります。

この段階で多くの方が困ってしまうことを理解しています。心配しないでください。

### シナリオ 1

私が提供したプロジェクト機能があなたのニーズに合っていることは分かっているが、調整方法が分からない場合。

この場合、私に直接メールを送って、あなたのニーズと「解決したいデータセット」を教えてください。私がモデルを微調整して、より良い結果を提供します。

料金はいただきませんが、時間の制約はありません。また、必ず実行することを保証するものではありません。（これが重要です！）

私はオープンソースプロジェクトを行っていますが、暇でやっているわけではありません。タイミングが合えば、モデルは更新されるでしょう。メールを送れば「可能性として」より良い結果が得られるかもしれません。それでも、双方にとって良い結果になるはずです。

### シナリオ 2

特定の機能を開発したいが、急いでいない場合。

それなら、私に連絡して一緒に議論しましょう。もし興味があれば、喜んで開発をお手伝いしますが、まずはある程度の規模のデータセットを準備してください。なぜなら、たとえ私が興味を持っていても、十分なデータを取得するための時間がない場合や、特殊なデータが必要であれば、特別な手段を取らなければならないからです。

このシナリオも同様に、料金はいただきませんが、時間に関する制約はありませんし、必ずしも実行することは保証できません。

:::tip
もし特定の機能が公開されているモデルコンペティション向けであれば、その場合はお受けできません。コンペティションには著作権や制限があることが多く、訴えられると運営元から問題を指摘されることがあります。
:::

### シナリオ 3

特定の機能の開発を迅速に行いたい場合。

時間が最優先事項であれば、開発委託として協力する方法に切り替えることができます。あなたのニーズに応じて、私が開発にかける時間を元に合理的な価格を提案します。

一般的に、プロジェクトの所有権は私が保持し、あなたはそれを自由に使用できます。プロジェクトの買い取りはお勧めしません。なぜなら、継続的な進化の理念に反するからです。技術が進歩するにつれて、今日の解決策はすぐに新しい方法に取って代わられる可能性があります。もしプロジェクトを買い取ったとしても、時間が経つとその投資の価値が失われるかもしれません。

:::tip
プロジェクトの所有権に関して理解できないかもしれませんが、考えてみてください。実際には「牛を飼う」より「ミルクを飲む」だけが目的なのかもしれません。

- 牛を飼うのはどれほど大変か？（エンジニアがプロジェクトを維持する必要がある）
- 場所を取って、世話が大変。（トレーニングマシンを準備する、クラウドマシンのレンタルは高く、サーバーは壊れやすい）
- 寒さと暑さに耐えられない。（モデルの調整に悩む）
- うまくいかないことがある。（期待した成果が得られない）
- 実際、かなり損している。（プロジェクトを買い取ってしまった）
  :::

さらに、多くのプロジェクトの最も価値のある部分はデータセットであり、次に解決策の考え方です。プライベートなデータセットをオープンソースにしない限り、コードを手に入れても最も大きな機能は観賞用に過ぎません。

もしよく考えた上で、それでもプロジェクトを買い取る決断をしたのであれば、止めはしません。どうぞ。
